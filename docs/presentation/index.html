Context

Building blocks

Two products this workbench brings together; each solves a different part of the LLM quality loop.

What is Promptfoo?

An open-source evaluation framework for LLM applications. You define prompts, test cases, and assertions in config; it runs your provider (e.g. OpenAI) in batch, grades outputs, and exports machine-readable results for CI, PRs, and regression checks.

  • Suite-based testing with repeatable scenarios and pass/fail (or custom) rules.
  • Typical for offline / pipeline use: same inputs every run so you can diff behavior over time.

What is Langfuse?

An LLM engineering platform for observability and evaluation. Your app (or eval jobs) send traces, generations, sessions, and scores; you inspect latency, cost, quality, and trends in a shared UI or API.

  • Central place for how the model is used in practice, not only whether a test case passed.
  • Supports scores, dashboards, and drill-down so product and eng align on the same quality picture.

Perspective 1

Problem space

What is hard?

  • Black-box LLMs — behavior shifts with prompts, models, and temperature.
  • No single metric — you need traces for debugging and aggregate scores for trends.
  • Manual chat ≠ CI — exploratory UI must pair with repeatable tests and exportable results.

What this project optimizes for

  • Observability — sessions, tags, and flushing traces to Langfuse during chat and benchmarks.
  • Comparable runs — benchmark JSON + Promptfoo configs + shared grading hooks.
  • Actionable dashboards — explicit score names you can filter in Langfuse (benchmark_*, promptfoo_*).

Perspective 1 — integration

Why a unified pipeline for Promptfoo and Langfuse?

Promptfoo answers “did the tests pass?” in CI; Langfuse answers “what happened, and how does quality trend?” for live and batched work. Wiring them together avoids two parallel stories about the same product.

One place for quality signal

  • Same vocabulary — offline eval scores sit next to session-scoped app and benchmark metrics instead of living in a separate spreadsheet or report.
  • Comparable over time — Promptfoo runs become first-class rows in Langfuse Scores, so regressions from pipeline changes show up where ops and eng already look.
  • Less context switching — no hand-off between “the eval tool” and “the observability tool” when you triage a drop in pass rate.

What each layer adds

  • Promptfoo — deterministic suites, assertions, and machine-readable output suited to automation and PR gates.
  • Langfuse — traces, sessions, tags, and dashboards for human analysis and stakeholder review.
  • The link — a short sync path from eval results to Langfuse keeps batch results actionable without re-implementing observability inside Promptfoo.

Perspective 1 — layers

Why keep both Promptfoo and Langfuse?

They are complementary, not redundant: one layer gates changes and holds benchmarks steady; the other watches real usage and how quality moves over time. Keep both in the stack.

Promptfoo

Offline · gates · suites

  • Regression testing — the same cases run again after every prompt, model, or app change.
  • CI — pass/fail signal in the pipeline, scriptable, machine-readable, PR-friendly.
  • Controlled benchmarks — fixed test sets and rubrics, so you know what you’re measuring.

Langfuse

Live · product · time series

  • Production evaluation — what the assistant actually does for users, not only in lab tests.
  • Observability — traces, sessions, and context to debug a bad outcome end to end.
  • Trend tracking — how latency, cost, and scores move as you ship; spot drift without running a new suite by hand every time.
  • Light scoring — app and eval scores in one place for dashboards, not a second analytics silo.

Rationale

Why use it? Why implement?

Value for people running evals vs. reasons to adopt or build this pattern in your own stack.

Why use it?

  • One loop — explore in chat, stress-test in the benchmark, and ship Promptfoo runs without switching tools for every task.
  • Faster debugging — Langfuse ties sessions, tags, and scores so you can go from a bad score to the trace, not a dead log file.
  • Regressions you can see — benchmark_* and promptfoo_* land in the same product you already use for monitoring.
  • Model comparison — two-model mode supports side-by-side behavior checks before you pick a default model.

Why implement (this design)?

  • Small, inspectable surface — Streamlit UI + a few Python modules; easy to change prompts, data, and scoring rules.
  • CI-friendly evals — Promptfoo writes JSON; a short sync script posts scores to Langfuse so pipelines stay scriptable.
  • Boring, portable stack — OpenAI + Langfuse + env-based config: the same skills transfer to other apps.
  • Tests as documentation — pytest around benchmark logic and sync gives safety when you extend the workbench.

Interpretation

Promptfoo scores in Langfuse

After the sync step, you read named scores in Langfuse tied to a session—this slide is a compact guide to that reading.

Two score types (from the sync script)

  • promptfoo_item_pass — numeric, usually 0 or 1; one per test row. 1 = that row passed Promptfoo’s grading; use for which cases failed.
  • promptfoo_accuracy — 0.0–1.0; one per eval upload. Value = passed rows ÷ total rows; the headline for the full run when the filter is one session.
  • Extras: comments and metadata (e.g. eval_id, test keys) help map scores back to the JSON and CI run.

Session & “tracing” in this path

Scores attach to a session_id you set via env, CLI, or the default promptfoo-{evalId} so one Promptfoo run = one session slice in the UI.

  • In Sessions / Scores — filter to that id to see one upload without mixing other runs.
  • Aggregated Scores view — averages and counts roll up across the time range; they mix many runs unless you narrow filters.
  • Dotted “tracing” in the app — Streamlit + langfuse.openai send live generations to Langfuse; Promptfoo in CI usually does scores only unless you also instrument those calls—same product, different path.

Mental model: what to open for which question

Question What to use
Did this full eval do well? promptfoo_accuracy for the correct session (and time window).
Which individual tests failed? promptfoo_item_pass and metadata; drill from Scores or session detail.
Is this comparable to the last run? Same session_id convention, same eval_id in metadata, and a clear baseline window.
What does a 0/1 pattern on item passes show? Per-row pass rate; the aggregate row should line up with promptfoo_accuracy for that run.

Headline = one run’s promptfoo_accuracy; debug = item passes + metadata. Always fix the session and date filter before comparing numbers.

Guidance

Pipeline best practices

Principles for credible evaluation and durable operations—not a checklist of tools.

What to do

  • Align evals with reality — measure the tasks and constraints that actually matter in production, not a toy that scores well.
  • Keep comparisons fair — change one thing at a time, or be explicit about what changed when you compare runs.
  • Unify signal — put interactive traces and batch scores in the same observability story so teams share one view of quality.
  • Govern like prod — credentials, data, and access deserve the same discipline as your shipping systems.
  • Act on both drill-down and roll-up — use aggregates for trends, dig into individual failures for root cause.

What to avoid

  • The single headline number — a pass rate without context, variance, or failure modes can mislead every stakeholder.
  • Orphaned automation — evaluations that never feed the same dashboards and reviews as live traffic.
  • Reward hacking the rubric — if the system optimizes the metric instead of the user outcome, the metric is wrong.
  • Silent change — shifting models, prompts, or policies without a record of what the baseline was.
  • Treating eval as a gate only — a pipeline that only blocks releases misses the point of learning why behavior changed.
